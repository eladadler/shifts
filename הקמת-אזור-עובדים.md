# הקמת אזור העובדים — מה צריך לעשות פעם אחת

אפליקציית העובדים (`עובד.html`) מוכנה, אבל כדי שהיא תעבוד צריך להשלים **שלושה צעדים חד-פעמיים** ב-Supabase. בלעדיהם העובדים יראו הודעת שגיאה.

## צעד 1: יצירת הטבלאות

1. היכנסו ל-[Supabase Dashboard](https://supabase.com/dashboard) → הפרויקט שלכם
2. בתפריט הצד: **SQL Editor** → **New query**
3. העתיקו את כל התוכן של הקובץ `supabase-setup.sql` שבריפו, הדביקו, ולחצו **Run**

זה יוצר שלוש טבלאות:

| טבלה | מי כותב | מי קורא | תפקיד |
|---|---|---|---|
| `app_employees` | המשבץ (אוטומטית) | אפליקציית העובד | זיהוי עובד לפי אימייל |
| `employee_requests` | העובדים | המשבץ | בקשות שיבוץ |
| `published_schedules` | המשבץ ("אישור ופרסום") | העובדים | סידורים שפורסמו |

## צעד 2: הפעלת התחברות ויצירת חשבונות לעובדים

1. בתפריט הצד: **Authentication** → **Sign In / Up** → ודאו ש-**Email** מופעל
   (מומלץ לכבות שם את "Allow new users to sign up" — אין הרשמה עצמית)
2. **Authentication** → **Users** → **Add user** → **Create new user**
3. לכל עובד: הזינו אימייל + סיסמה זמנית, וסמנו **Auto Confirm User**
4. מסרו לכל עובד את האימייל והסיסמה הזמנית. הוא יוכל להחליף סיסמה דרך "שכחתי סיסמה" באפליקציה.

## צעד 3: שיוך אימיילים לעובדים באפליקציית המשבץ

1. פתחו את אפליקציית המשבץ (`מערכת שיבוץ.html`) → תפריט ☰ → **עובדים**
2. אצל כל עובד: **עריכה** → מלאו את שדה **אימייל** — בדיוק אותו אימייל שיצרתם לו בצעד 2
3. שמרו. הרשימה מסתנכרנת אוטומטית לענן.

## זהו! איך זה עובד מעכשיו

- **פרסום סידור:** לחיצה על "אישור ופרסום" אצל המשבץ מפרסמת את הסידור לעובדים. "חזרה לעריכה" מסירה את הפרסום.
- **בקשות:** עובד מסמן ימים ושולח מהאפליקציה שלו. אצל המשבץ הבקשות מופיעות במסך "בקשות" עם תגית "📥 הוגש מאפליקציית העובד" (רענון אוטומטי כל דקה, או כפתור רענון ידני).
- עובד יכול לעדכן את הבקשות שלו עד שהסידור לאותו חודש מתפרסם — ואז הן ננעלות לקריאה בלבד.

## איך העובדים ניגשים לאפליקציה?

`עובד.html` הוא קובץ עצמאי — צריך לארח אותו בכתובת שנגישה מהטלפון. האפשרות הפשוטה: להפעיל GitHub Pages על הריפו (Settings → Pages → Deploy from branch → main), ואז הקובץ יהיה זמין בכתובת קבועה שאפשר לשלוח לעובדים. שימו לב שהריפו ציבורי במקרה כזה — אפשרות מאובטחת יותר היא אירוח ב-Netlify/Vercel.

> **הערת אבטחה:** ההרשאות בטבלאות פתוחות (כמו הטבלה הקיימת של המערכת) — מי שמחזיק במפתח ה-publishable יכול לקרוא ולכתוב. ההתחברות באפליקציית העובד היא שכבת זיהוי ונוחות, לא חומת אבטחה. לצוות קטן זה סביר; אפשר להדק בעתיד עם מדיניות RLS מבוססת-אימייל.
